# Changelog

All notable changes are recorded here. The build system combines the developer-owned
Major.Minor with a CI build number to form Major.Minor.Build.

## Next Version

- Added an **Azure DevOps** tab to the options page and moved the read-only ADO Configuration panel (organization and project) there from Appearance, which now keeps only Theme and Default view. The new tab adds three synced-per-device settings: a searchable **Current team** picker populated from the detected organization/project, a **Future sprints** count (a whole number from 1 to 12, offered in the sprint picker in addition to the current sprint), and an **Area paths** list where each row has an area-path field with autocomplete (matching any part of the path) plus a label that defaults to the path's last segment, with controls to add and delete rows. Each setting saves as you change it.
- Added a **Work item types** mapping table at the end of the Azure DevOps options tab. Its columns are your team's own application states (user-defined — rename, remove, or add up to six), and each row is one work item type, picked from a searchable list of the organization/project's types shown with their Azure DevOps icon and color. In each cell you add that type's ADO states from a searchable field; a state can sit in only one column, and the first state in a cell is highlighted as that column's primary value (the one written back to ADO). Any ADO state you do not place falls back to the first column (treated as "not picked up"). Columns, rows, and their choices sync across your devices and save as you edit.
- The extension now treats itself as configured only once the Azure DevOps settings are complete (a current team, at least one area path, at least one board column, and at least one work item type with a mapped state). While a query is bound but the configuration is incomplete, the enhanced view is held back and the query stays on Azure DevOps' own view, the options page shows an error banner explaining why, and the AwesomeADO button's menu hides the view-swap options until you finish the setup.
- The Azure DevOps options tab now loads the detected organization's **teams** and **area paths** using your signed-in Azure DevOps session, so the **Current team** picker and **Area paths** autocomplete offer real suggestions with no extra sign-in or permission prompt.
- The Query Bindings tab now splits into two sections once a query is bound: the Query bindings card keeps the bound-query dropdown, query ID, name, and Delete, while a new Query View Configuration card holds the enhanced-view picker and Save. While binding a new query everything stays in the single Query bindings card (view picker and Save included) and Delete is hidden, since there is nothing to remove yet.
- The Query Bindings tab now adapts to how it is opened. Starting a bind from a query's AwesomeADO button shows that query's ID and name (read-only) with the enhanced-view picker and its configuration, and hides the bound-query dropdown until you Save; after saving, the dropdown appears with the just-bound query selected. Opening the tab from the Options menu shows the bound-query dropdown preselected to the query on the Azure DevOps tab you came from (falling back to the first bound query) along with its read-only ID and name, so you can edit or remove that binding. New bindings are created from a query's own AwesomeADO button rather than from a list of open tabs.
- Refined the **Work item types** table on the Azure DevOps options tab: a work item type's searchable picker now appears only while adding a new row and collapses to a read-only colored label once you choose the type (remove the row to change it), each row's remove control is now the same compact **×** used for columns, the state field's suggestions reopen immediately after you add a state so you can keep adding without clicking away, and the type/state suggestion lists now float above the table instead of being clipped inside its scroll area (flipping above the field when there is no room below). The options page also uses smaller, window-relative side margins so widening the browser window gives the settings more room.
- Further polished the **Work item types** table: the type picker no longer offers a type already used by another row, a row's per-column add-state fields stay hidden until you pick its type and disappear again once every one of that type's states is mapped, the work-item-type column is narrower, and the ADO state chips in each column can be dragged to reorder them — with the first (primary/default) chip highlighted so it is clear which value gets written back to ADO.
- A fresh install now seeds a fifth board column, **Removed**, after Queue/Active/Waiting/Done, and long work item type names in the type picker show a tooltip with the full name when they are cut off.
- The **Diagnostics** options tab now shows a device-local activity log: the extension records what it does and, in particular, any errors it hits, each line stamped with a timestamp and ordered oldest to newest. A quick **Errors only** filter hides routine informational lines so you can focus on problems, an **Export…** button downloads the shown lines as a timestamped text file, and **Clear** empties the log. The log stays on this device only — it is never synced to your other browsers or sent anywhere.
- Diagnostics log lines are now tagged with the specific part of the extension that wrote them — the feature area (component folder) that owns the code, for example `content/query-page`, `common/settings`, or `options/alerts`, shown in each line — instead of a single broad area name. The Diagnostics tab replaces the earlier component checkboxes with a searchable **Sources** dropdown: type to filter the list of sources, tick one or more to show only those, and use **Select all** / **Clear all** to reset. The extension also records more of its decisions — when it enhances a query page or leaves it on Azure DevOps' own view (with the reason and the signals behind it), when settings or query bindings are saved (names only, never values), and when the top-bar button and menu appear or change — so the log explains not just what happened but why. The AwesomeADO button's menu gains a **View Log** item at the bottom that opens the options page straight to the Diagnostics tab.
- Opening the options page from the AwesomeADO button's menu now reuses an already-open options tab instead of stacking up duplicates: **View Log** and the other menu items focus the existing options tab and switch it to the requested section in place (so **View Log** always lands on Diagnostics even when the options page is already open on another tab), only opening a new tab when none is already open.

## 0.1 - Initial scaffolding and basic functionality

- Initial release of the AwesomeADO MV3 extension for Chrome and Edge, including CI/Release pipelines with an immutable release workflow and changelog-validated versioning, and the AwesomeADO icon (an Azure DevOps work-item card with a checkmark and sparkle) shown in the browser toolbar and extensions list.
- Added per-query bindings: each Azure DevOps query can be bound to a view (Sprint View or Project Tracking) with its own settings, so the same view can be configured differently per query. Bindings sync across your devices.
- Added an "Enhance with AwesomeADO" button to the Azure DevOps top bar that appears on any saved query page (bound or not) and opens a popup menu aligned under it. The button shows the extension icon (its purpose stays available as a tooltip and to screen readers) and sits inside the Azure DevOps top-bar command bar alongside ADO's own icons, styled to match them (flat square button with the same hover highlight). On an unbound query the menu offers Options and "Enable Enhanced View" (which opens the options page to bind that query); on a bound query the menu lets you switch that query between its bound view (e.g. Sprint View) and ADO's Standard View, open Options, or "Disable Enhanced View" to remove the binding. The per-query view choice is remembered, synced across your devices, and overrides the global default view.
- Unbound Azure DevOps queries are always left on ADO's own view — the extension only injects its button and never replaces the page until you bind the query. The global Default view applies only to bound queries, and query-page enhancement hides only the content below the top breadcrumb bar, keeping ADO's breadcrumb navigation visible.
- Themeable, tabbed options page (Appearance, Query Bindings, and a Diagnostics placeholder), with a Theme selector (Auto/Light/Dark/Blue, where Auto follows Azure DevOps' own theme), a read-only ADO Configuration panel showing the active tab's organization and project, and a Default view selector (Original ADO View / Enhanced View). The options page header shows the AwesomeADO icon.
- The Query Bindings tab manages per-query view bindings: open it from a query's AwesomeADO button to bind that query (its name is shown read-only), or open it from the button's Options menu to pick from the queries you have open (and any already-bound queries) in a dropdown. Each query can be given an Enhanced View Type, saved, or deleted (unbound), so several queries can each be managed independently.
- The options page detects open ADO Query tabs regardless of which tab is focused, so opening the options tab in the foreground no longer hides the active query's organization and project. It also shows a visible error message when it fails to load or hits an error later, instead of failing silently.
- Auto theme reads Azure DevOps' theme color token instead of the page's white loading fallback, so a Dark ADO account is no longer misread as Light (no options-page white flash).
- The button's "Options" and "Enable Enhanced View" menu items reliably open the options page; failures to open it are now reported instead of being silently ignored.
